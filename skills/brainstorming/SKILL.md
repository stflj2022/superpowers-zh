# 头脑风暴：将想法转化为设计

## 概述

通过自然的协作对话，将想法转化为完整的设计和规格。

首先了解当前项目上下文，然后一次问一个问题来完善想法。一旦你理解了要构建的内容，分小段展示设计（每段 200-300 字），每段后检查是否正确。

## 流程

**理解想法：**
- 首先查看当前项目状态（文件、文档、最近提交）
- 一次问一个问题来完善想法
- 尽可能使用选择题，但开放式问题也可以
- 每条消息只问一个问题 - 如果一个主题需要更多探索，将其分解为多个问题
- 重点关注：目的、约束、成功标准

**探索方案：**
- 提出 2-3 种不同方案及其权衡
- 以对话方式展示选项，给出建议和理由
- 首先展示你推荐的选项并解释原因

**展示设计：**
- 确认你理解了要构建的内容后，展示设计
- 将其分解为 200-300 字的小节
- 每节后询问目前看起来是否正确
- 覆盖：架构、组件、数据流、错误处理、测试
- 准备在有些内容不合理时回去澄清

## 设计完成后

**文档化：**
- 将验证后的设计写入 `docs/plans/YYYY-MM-DD-<主题>-design.md`
- 如果有 elements-of-style:writing-clearly-and-concisely 技能，使用它
- 将设计文档提交到 git

**实施（如果继续）：**
- 询问："准备好设置实施了吗？"
- 使用 superpowers-zh:using-git-worktrees 创建隔离工作空间
- 使用 superpowers-zh:writing-plans 创建详细实施计划

## 核心原则

- **一次一个问题** - 不要用多个问题压倒用户
- **优先使用选择题** - 尽可能比开放式问题更容易回答
- **严格执行 YAGNI** - 从所有设计中删除不必要的功能
- **探索替代方案** - 在确定之前总是提出 2-3 种方案
- **增量验证** - 分段展示设计，逐段验证
- **保持灵活** - 当有些内容不合理时回去澄清
